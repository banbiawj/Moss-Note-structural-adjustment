# Canvas Paging Tools Design

## Goal

Add first-version canvas paging tools that let the agent read blocks before or after the current reading context, then merge those blocks into the current `canvas_context` in `canvas_snapshot` order.

## Scope

V1 includes:

- `canvas_read_before`
- `canvas_read_after`
- structured context block metadata
- ordered merge by snapshot index
- block de-duplication by `moss-block-*` id
- prompt rendering from structured context
- explicit gap markers for unloaded ranges

V1 does not include:

- token budget enforcement
- automatic compaction
- summary compression
- pinning
- section jumping
- adding search results to context
- frontend visualization of reading history

The only guardrail in V1 is a per-call `block_count` clamp so one tool call cannot pull an unbounded amount of HTML.

## Current Facts

`moss-block-*` ids are stable anchors, not order keys. They are generated by the frontend editor and by backend file import. Production ids are random, so ordering must come from the parsed order of blocks in `canvas_snapshot`.

The backend already parses top-level Moss blocks from `canvas_snapshot`. That parsing order is the source of truth for forward and backward paging.

Current `AgentTask.canvas_context` is a string. V1 keeps this field as the prompt-facing value and adds an internal structured block list on the task.

## Architecture

Paging tools return structured block deltas. They do not mutate prompt text directly.

Flow:

```text
LLM calls canvas_read_after or canvas_read_before
  -> tool reads blocks from canvas_snapshot
  -> tool returns JSON operation canvas_context_add
  -> tools_node merges returned blocks into task canvas_context_blocks
  -> merge sorts by snapshot index and removes duplicates
  -> renderer rebuilds task canvas_context string
  -> task_prompt is rebuilt with the new canvas_context
  -> LLM continues with the expanded context
```

## Tool Semantics

`canvas_read_before(anchor_block_id?: string, block_count: int = 3)`

- If `anchor_block_id` is provided, use it.
- Otherwise use the minimum-index block already in `canvas_context_blocks`.
- If context has no blocks, use `focus_block_id`.
- Return up to `block_count` blocks before the anchor.
- Return blocks in snapshot order.

`canvas_read_after(anchor_block_id?: string, block_count: int = 3)`

- If `anchor_block_id` is provided, use it.
- Otherwise use the maximum-index block already in `canvas_context_blocks`.
- If context has no blocks, use `focus_block_id`.
- Return up to `block_count` blocks after the anchor.
- Return blocks in snapshot order.

`block_count` is clamped to `1..8` in V1.

## Context Block Shape

Each block in the structured context should include:

```json
{
  "block_id": "moss-block-abc",
  "index": 12,
  "tag": "p",
  "heading_path": ["Section"],
  "text": "Visible text",
  "html": "<p id=\"moss-block-abc\">Visible text</p>",
  "source": "initial | read_before | read_after",
  "added_at": 1
}
```

`index` is the only order key. `added_at` is kept for later compaction design but does not affect rendering order.

## Merge Rules

Merge behavior:

- Existing and newly read blocks are keyed by `block_id`.
- If a block already exists, keep the existing item.
- If a block is new, insert it.
- Sort the merged list by `index`.
- Render the merged context into prompt text.

Unloaded gaps are marked in the rendered string:

```text
[Omitted blocks 13-29]
```

This prevents the model from treating sparse context as continuous source text.

## Acceptance Criteria

- Calling `canvas_read_after` twice without an explicit anchor advances from the current highest index.
- Calling `canvas_read_before` after reading forward inserts earlier blocks before later blocks in rendered context.
- Re-reading overlapping blocks does not duplicate them.
- Random-looking `moss-block-*` ids are never sorted lexically.
- `task_prompt` uses the newly rendered `canvas_context` after paging.
- Existing `search_document_blocks`, `update_canvas_element`, and `generate_download_link` behavior remains intact.
